\section{Implementation}
The implemented project focuses on the partition phase of a Partitioned Hash Join under duplicate-heavy workloads.

\subsection{Execution Variants}
The codebase provides a reproducible benchmarking pipeline to compare three execution variants: \texttt{plain\_novec}, 
\texttt{plain\_vec}, and \texttt{avx2}. Each variant is got by a different compilation of the same file (\texttt{main.cpp}) 
and applies the same high-level algorithm but uses different optimization strategies.

The executable targets can be built with the commands: \texttt{make plain\_novec}, \texttt{make plain\_vec} and \texttt{make avx2}. 

Executing the programs as defined in the \texttt{README.md} file, they receive the same command-line arguments, which include:
\begin{itemize}
    \item the dataset size \texttt{N},
    \item the number of partitions \texttt{P},
    \item the name of the hash function to be used,
    \item the output CSV file path. 
\end{itemize}
The runtime workflow includes: loading the two input datasets, partitioning both relations with the 
selected hash function, computing throughput and checksums, writing partitioned binary outputs and appending one result row 
to the CSV file. This row contains the full experiment metadata (\texttt{N}, \texttt{P}, hash, execution type), timing metrics 
(partition time, global time) and other fields (like throughput and checksum), which are then used by the analysis notebooks 
to extract statistics.

\subsection{Grid Search approach}
The experimental automation is implemented with Bash orchestrators and Slurm runners. The script \texttt{run\_grid\_search.sh} performs:
\begin{itemize}
    \item cleanup (\texttt{make cleanall}) and compilation (\texttt{make}),
    \item dataset generation job submission,
    \item chained Slurm submissions with dependencies for \texttt{plain\_novec} $\rightarrow$ \texttt{plain\_vec} $\rightarrow$ \texttt{avx2}.
\end{itemize}

The search space is defined in \texttt{runners/grid\_config.sh}: \texttt{N} $\in \{10^5, 10^6, 10^7\}$, \texttt{P} $\in \{16,\dots,4096\}$ (powers of two), 
hash $\in \{\texttt{mask}, \texttt{xorshift}, \texttt{fmix32fold}\}$. This gives 81 configurations per executable for each full sweep.

The script \texttt{stability\_benchmark.sh} uses \texttt{runners/grid\_config\_2.sh} (\texttt{N} fixed to $10^7$) and repeats the same parameter grid \texttt{M} 
times (in the current dataset, 10 repetitions). This is used to separate raw performance from run-to-run variance.

An important implementation detail is that \texttt{compute\_partitions\_avx2} currently supports only \texttt{mask}. For unsupported hashes (\texttt{xorshift}, 
\texttt{fmix32fold}), the AVX2 executable exits early after the runtime check. Therefore:
\begin{itemize}
    \item \texttt{results/results.csv} contains 81 rows for \texttt{plain\_novec}, 81 rows for \texttt{plain\_vec}, and 27 rows for \texttt{avx2};
    \item \texttt{results/benchmark\_results.csv} is used for stable repeated-run analysis.
\end{itemize}

This design makes experiments reproducible and auditable: build flags, execution logs, and vectorization diagnostics are persisted and tied to each benchmark batch.

\subsection{Choice of the number of partitions \texttt{P}}
The implementation enforces \texttt{P} as a power of two (runtime check in \texttt{main.cpp}). This constraint enables partition-id computation with:
\[
\texttt{part\_id} = \texttt{hash} \ \& \ (\texttt{P}-1),
\]
which is used in all selected kernels and avoids modulo/division in the hot loop. In practice, this gives a predictable low-cost mapping and a SIMD-friendly pattern.

The explored range \texttt{P} $\in \{16,32,\dots,4096\}$ spans 4 to 12 partition bits. It is large enough to test coarse-to-fine partitioning effects while keeping the full grid-search cost manageable.

Observed results suggest that, inside this range, \texttt{P} has a smaller impact than hash complexity: for fixed \texttt{exec\_type} and hash, mean partition time varies by less than about 1\% across \texttt{P}, while switching hash changes absolute cost much more (e.g., in \texttt{plain\_novec}: \texttt{mask} $\approx 0.01837$ s, \texttt{xorshift} $\approx 0.02917$ s, \texttt{fmix32fold} $\approx 0.04391$ s).

\subsection{Hash Functions}
\textbf{Mask hashing}\\
This is the simplest mapping. It does not really mix the bits of the key: it just keeps the lowest bits by applying a bitwise AND with P - 1.
So, logically, it is just a bit selection operation: the partition id is obtained from the least significant bits of the input key.
\\\textbf{Xorshift hashing}\\
This function performs a sequence of right shifts and XORs on the 64-bit key.
The idea is to spread information from higher bits into lower bits, so that the final low bits used for partitioning depend on more than just the original least significant bits.
\\\textbf{Fold32 + fmix32 hashing}\\
This function first folds the 64-bit key into 32 bits by XORing the lower 32 bits with the upper 32 bits.
Then it applies the fmix32 finalizer, which combines XORs, shifts and 32-bit multiplications to further scramble the bits.
So, it first compresses the 64-bit value into 32 bits, then performs a stronger bit diffusion so that small differences in the input affect many output bits.

\subsection{Correctness verification}
Correctness is validated with checksum-based comparisons: each run computes a checksum over the partitioned outputs and stores it in CSV, so implementations can be compared without printing full arrays.

All optimized variants were checked against the plain reference (\texttt{plain\_novec}) over the benchmark grid. In both \texttt{results/results.csv} and \texttt{results/benchmark\_results.csv}, every comparable group has a single unique checksum.

In addition, for one small input size (\texttt{N}=$10^5$), an explicit element-by-element comparison was performed between the plain output and an optimized output, confirming exact equality.


\section{SIMD Optimization}
\subsection{Compiler evidence of vectorization}
Vectorization is checked through GCC diagnostics generated at build time. In the Makefile, \texttt{plain\_vec} and \texttt{avx2} are compiled with:
\texttt{-fopt-info-vec-optimized -fopt-info-vec-missed}, while \texttt{plain\_novec} is compiled with: \texttt{-fno-tree-vectorize}.

The following lines in \texttt{compilation/gcc\_vec\_report.txt} prove auto-vectorization for the \texttt{plain\_vec} build for each hash function:
\begin{verbatim}
lib/partition.cpp:9:31: optimized: loop vectorized using 32 byte vectors
lib/partition.cpp:9:31: optimized: loop vectorized using 16 byte vectors
lib/partition.cpp:20:31: optimized: loop vectorized using 32 byte vectors
lib/partition.cpp:20:31: optimized: loop vectorized using 16 byte vectors
lib/partition.cpp:36:31: optimized: loop vectorized using 32 byte vectors
lib/partition.cpp:36:31: optimized: loop vectorized using 16 byte vectors
\end{verbatim}
and the manual vectorization in the AVX2 path:
\begin{verbatim}
lib/partition_avx2.cpp:36:14: optimized: loop vectorized using 16 byte vectors
lib/partition_avx2.cpp:21:27: missed: couldn't vectorize loop
\end{verbatim}
The missed one is the loop of the manual AVX2 implementation, the other is the loop for the tail of the data. This is expected, because the compiler 
does not recognize the custom intrinsics as a vectorizable pattern.

During development, initial 64-bit multiply-based mappings were also tested. GCC auto-vectorized those loops too, but they did not produce profitable end-to-end performance. The final selected hashes (\texttt{mask}, \texttt{xorshift}, \texttt{fmix32fold}) rely on simpler bitwise and 32-bit-friendly operations, which are more suitable for efficient auto-vectorization on the target CPU.

\subsection{AVX2 mask hash implementation}
The AVX2 partition kernel is implemented in \texttt{lib/partition\_avx2.cpp}. The logic is the following:
\begin{enumerate}
    \item Build SIMD constants: \texttt{mask\_vec = broadcast(P-1)} and \texttt{perm\_idx = [0,2,4,6,...]}.
    \item Process four 64-bit keys per iteration (\texttt{\_\_m256i} register):
    \begin{itemize}
        \item load 256 bits with \texttt{\_mm256\_loadu\_si256},
        \item compute partition ids via bitwise AND with \texttt{\_mm256\_and\_si256},
        \item pack low 32-bit lanes with \texttt{\_mm256\_permutevar8x32\_epi32},
        \item store 4 contiguous \texttt{uint32\_t} outputs with \texttt{\_mm\_storeu\_si128}.
    \end{itemize}
    \item Handle the tail (\(< 4\) keys) with scalar code to preserve correctness for any input length.
\end{enumerate}

\subsection{Limits analysis}
The observed ranking (\texttt{mask} $>$ \texttt{xorshift} $>$ \texttt{fmix32fold} in absolute throughput) and speedups indicate different limiting factors across kernels.

For \texttt{mask}, arithmetic intensity is minimal (load + bitwise AND + store), and measured SIMD gains are relatively modest (\(\approx 1.11\times\) for \texttt{plain\_vec}, \(\approx 1.25\times\) for \texttt{avx2} over \texttt{plain\_novec}). This is consistent with a kernel that is closer to memory-bound behavior: computation is cheap, so memory traffic and write bandwidth limit the maximum gain.

For \texttt{xorshift}, extra shift/XOR instructions increase compute work and still vectorize well, giving a larger gain (\(\approx 1.31\times\)) but still below \texttt{fmix32fold}. For \texttt{fmix32fold}, two 32-bit multiplications and additional mixing steps raise arithmetic cost substantially; scalar throughput is lowest, but SIMD has more compute to amortize and reaches the highest relative gain (\(\approx 1.75\times\)). So, increasing hash complexity lowers absolute throughput but can increase relative SIMD speedup when the kernel becomes more compute-bound.

Fixed overheads are also visible at smaller \texttt{N}: from \texttt{results/results.csv} for \texttt{mask}, throughput increases from about \(9.95\times 10^8\) elems/s at \texttt{N}=$10^5$ to \(1.09\times 10^9\) at \texttt{N}=$10^7$ in \texttt{plain\_novec} (similar trend in \texttt{plain\_vec} and \texttt{avx2}). This supports the interpretation that setup and non-kernel costs matter more for small inputs and are progressively amortized.


\section{Conclusions}
The experiments confirm that partition performance is shaped by both algorithmic choices (\texttt{hash}, \texttt{P}) and execution strategy (\texttt{plain\_novec}, \texttt{plain\_vec}, \texttt{avx2}), with different factors dominating in different regions.

Hash selection has the strongest effect on absolute cost: \texttt{mask} is consistently the fastest, \texttt{xorshift} is intermediate, and \texttt{fmix32fold} is the most expensive due to stronger mixing.

 SIMD still helps, but gains are not uniform. In the repeated benchmark, \texttt{plain\_vec} improves over \texttt{plain\_novec} by about $1.11\times$ on \texttt{mask}, $1.31\times$ on \texttt{xorshift}, and $1.75\times$ on \texttt{fmix32fold}. The manual AVX2 kernel gives the best \texttt{mask} throughput (\(\approx 1.36 \times 10^9\) elems/s), with about $1.25\times$ speedup over scalar.

The number of partitions \texttt{P} still affects runtime, but in these results its impact is generally smaller than the choice of hash and execution kernel. At the same time, AVX2 support is currently limited to \texttt{mask}, so the fastest path is not yet available for the other hash functions.
